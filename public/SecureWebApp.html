Secure Scheduling Web App
Overview

Online scheduling systems are essential to the day-to-day operations of businesses across all industries.

This project involved designing, developing, and testing a secure web application with robust protection against the common web threats detailed in the UK Cyber Essentials standard and OWASP Top 10 such as XSS, CSRF, SQL Injection, and authentication vulnerabilities.

The web app is a secure scheduling solution aimed for small businesses. It is built using modern web technologies such as the Laravel framework and follows best practices in secure coding, input validation, and session management.

Technologies

Development:

PHP Laravel framework, MySQL database, XAMPP Web Server, HTML, CSS, Tailwind, JavaScript, Lucid.

Testing:

VirtualBox VM, Kali Linux, Burp Suite, SQL Map, Inetsim dummy web-server.
Security Testing

This section outlines a few of the penetration tests performed.

On Kali Linux, Inetsim was used to simulate internet services. Inetsim is a powerful tool that allows internet services to be simulated, acting as a fake or dummy server to capture network activity. The web application is limited to localhost and not exposed to the internet which makes it suitable for hosting during security testing. This is important as only this web application will be affected if the server crashes or becomes corrupted during security testing.

Brute-force

Burp Suite was used to perform a Sniper attack on the login page, which is a type of brute-force attack that attempts to guess the password by trying every possible combination. It is a powerful tool for web application security testing, allowing for the inspection and manipulation of web traffic.

The attack was unsuccessful as the login POST route restricts users to a maximum of 5 login attempts before they are automatically locked out of the login screen for 1 minute, successfully preventing brute-force attacks.

Cross-Site Request Forgery (CSRF) attack

CSRF tokens are in place at all input fields for editing or changing information, a unique, secret, and unpredictable value that is generated by the server and included in each request made by the client. This token is used to verify that the request is coming from a legitimate user and not an attacker.

SQL Injection

Another test was performed using both manual testing and the SQLMap tool to test for SQL injection vulnerabilities.

URL encoding was also attempted, which encodes the apostrophe in URL encoding format %27. The payload ‘test%40example.com%27%20OR%201%3D1%20—’ was also rejected by the input validation.

Input sanitization and validation protected the web app against SQL Injection attacks as shown above.
